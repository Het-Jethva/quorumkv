# Persist consensus state in a purpose-built WAL

Each node will persist Raft hard state (current term, vote, and commit index), log appends and conflict truncations, snapshot metadata, and stable cluster and node identities in a minimal append-only, framed, checksummed, segmented write-ahead log. Every durable file carries a magic value and explicit format version; v1 refuses unsupported versions rather than guessing or migrating them automatically. Snapshots are written and synced as temporary files, then renamed to unique immutable filenames containing their included index and term; startup validates candidates and selects the newest snapshot compatible with the WAL rather than overwriting the last good snapshot in place. The live key-value state will remain in memory and will be restored from that snapshot plus committed WAL entries through the durable commit index; `lastApplied` will not be persisted separately. There will be no second persistent representation to keep atomically synchronized. A dedicated WAL writer may group multiple ordered appends into one disk sync, but it may report persistence completion only after that sync succeeds, and production mode will never acknowledge unsynced records. After a new snapshot is durably installed, compaction deletes only complete WAL segments fully covered by that snapshot, retaining the active segment and all later history. QuorumKV will not embed a general-purpose database for consensus storage: the custom format increases the recovery-testing burden, but makes durability ordering, torn-write handling, replay, and compaction explicit without expanding into a general storage-engine project.
